01

Workflow individuel

Quand tu travailles seul sur ton dépôt. On commence par la mise en place, puis le cycle se répète tous les jours.

première fois sur le projet setup
# initialiser + lier au repo distant
git init
git remote add origin <url-du-repo>

git add .
git commit -m "Premier commit"
# -u : à ne faire qu'une fois
git push -u origin main
ensuite, chaque jour daily
# voir ce qui a changé
git status

# tout préparer…
git add .
# …ou un seul fichier
git add fichier.java

git commit -m "feat: ..."
git push

03

Workflow d'équipe — le flux complet

Règle d'or : jamais de commit direct sur dev ni main. Chacun travaille sur sa branche, puis passe par une Pull Request.
lier le projet au repo distant — 1re fois clone
# récupérer le projet + son historique (origin est déjà prêt)
git clone <url-du-repo>
cd projet
1
Synchroniser la branche dev
update dev
git switch dev
git pull origin dev
2
Créer sa branche
branch feature
git switch -c feature/ma-tache
3
Travailler et créer un commit
commit work
git add mes-fichiers
git commit -m "feat: ma tâche"
4
Avant de pousser : récupérer la dernière version du dev
rebase sync
# l'étape qui évite les conflits
git fetch origin
git rebase origin/dev

# si un conflit apparaît :
# 1. résoudre dans le fichier
git add fichier-resolu
git rebase --continue
5
Pousser sa branche → Pull Request
push + PR github
git push -u origin feature/ma-tache
# puis sur GitHub : ouvrir une
# Pull Request  feature/ma-tache → dev
6
Après validation, la Pull Request est fusionnée dans dev
maj dev merged
# tout le monde récupère le nouveau dev
git switch dev
git pull origin dev
7
Équipe d'accord : dev → main (mise en production)
release main
# réservé au lead / responsable
git switch main
git merge dev
git push origin main
Option : on peut aussi faire la fusion dev → main via une Pull Request, pour garder une trace de chaque mise en production.
05

Résoudre un conflit de fusion

Un conflit arrive quand deux branches modifient le même endroit d'un fichier. Git interrompt temporairement l'opération et marque les zones en conflit ainsi :

ce que Git écrit dans le fichier conflit
<<<<<<< HEAD
ta version (branche actuelle)
=======
version entrante (autre branche)
>>>>>>> branche-entrante
cas d'un merge commit
# 1. éditer : garder le bon contenu,
#    retirer <<< === >>>
git add fichier-resolu

# 2. valider la résolution
git commit -m "Résolution du conflit"
cas d'un rebase continue
# 1. éditer + retirer les marqueurs
git add fichier-resolu

# 2. reprendre le rebase
git rebase --continue

# tout annuler :
git rebase --abort
